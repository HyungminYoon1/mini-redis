# Server And CLI

## 서버의 역할

`internal/server/server.py`는 전체 오케스트레이터다.

하는 일:

- 소켓 생성
- bind / listen
- 연결 수락
- 세션 핸들러 생성
- 백그라운드 expiration sweeper 실행

즉, 실제 명령 의미를 아는 파일이 아니라 "시스템을 연결하는 파일"이다.

## 세션 핸들러의 역할

`internal/server/session_handler.py`는 연결 하나를 책임진다.

핵심 루프:

1. RESP 프레임 읽기
2. parse / validate
3. `HELLO 3`이 먼저 왔는지 확인
4. 명령 실행
5. 응답 전송

여기서 `command_lock`을 쓰는 이유는 저장소 변경을 단순하게 직렬화하기 위해서다.  
연결은 여러 개 받을 수 있지만, 명령 실행은 한 번에 하나씩 처리하도록 한 셈이다.

이건 Redis의 실제 구현과 같지는 않지만, 학습용으로는 concurrency보다 파이프라인 이해가 더 중요하므로 의도적인 선택이다.

## CLI의 역할

`cmd/mini_redis_cli/main.py`는 사람이 쓰는 인터페이스다.

흐름:

1. 서버 연결
2. `HELLO 3` 전송
3. 사용자 명령 전송
4. 응답 렌더링
5. 종료 코드 반환

CLI가 중요한 이유는 "같은 RESP 로직을 서버와 클라이언트가 공유한다"는 점을 보여주기 때문이다.

서버만 구현하면 프로토콜의 절반만 이해한 것이다.  
CLI까지 구현해야 네트워크 계약이 양방향이라는 사실이 보인다.

## 응답 렌더링

CLI는 내부 RESP 값을 그대로 출력하지 않고, 사람이 읽기 쉬운 문자열로 바꾼다.

예:

- simple string -> `OK`
- blob string -> 실제 문자열
- null -> `(nil)`
- error -> stderr에 `ERR ...`
- map -> `key: value` 형식

이 계층이 있는 이유는 "프로토콜 표현"과 "사용자 출력"을 분리하기 위해서다.

## 초보자가 여기서 꼭 이해해야 할 것

- 서버는 프로토콜과 비즈니스 로직을 연결하는 조립자다
- 세션 핸들러는 연결 하나의 라이프사이클을 관리한다
- CLI는 서버와 똑같은 프로토콜 계약을 이해해야 한다
