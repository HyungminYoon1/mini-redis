# 미니 Redis 협업 공통 규칙 문서

## 1. 문서 목적

이 문서는 미니 Redis를 여러 명이 병렬로 개발할 때 공통으로 따라야 하는 네이밍 규칙, 계층 규칙, 테스트 규칙, 협업 절차를 정의한다.

본 문서는 [`architecture.md`](architecture.md), [`api-spec.md`](api-spec.md), [`implementation-plan.md`](implementation-plan.md), [`team-work-allocation.md`](team-work-allocation.md)를 보완하며, 코드 스타일과 협업 방식의 차이로 인한 충돌을 줄이는 것을 목적으로 한다.

## 2. 기본 원칙

- 문서에 정의된 아키텍처와 API 계약을 우선한다.
- 한 파일에는 하나의 주된 책임만 둔다.
- 계층 경계를 넘는 편의 구현을 허용하지 않는다.
- 병렬 작업 중에는 파일 소유권과 인터페이스 합의를 우선한다.
- 공통 규칙을 바꿔야 할 경우 먼저 문서를 수정한 뒤 코드를 수정한다.

## 3. 네이밍 규칙

### 3-1. 파일명

- 모든 파이썬 파일은 `snake_case.py`를 사용한다.
- 파일명은 역할이 드러나야 한다.
- 축약어는 꼭 필요한 경우만 사용한다.

예시:

- `request_decoder.py`
- `response_encoder.py`
- `expiration_sweeper.py`
- `runtime_config.py`

### 3-2. 클래스명

- 클래스명은 `PascalCase`를 사용한다.
- 클래스명은 역할 중심으로 짓는다.
- 인터페이스 성격의 추상 클래스도 동일하게 `PascalCase`를 사용한다.

예시:

- `ExpirationManager`
- `ExpirationSweeper`
- `RequestDecoder`
- `StoreRepository`

### 3-3. 함수 및 메서드명

- 함수와 메서드명은 `snake_case`를 사용한다.
- 동작이 드러나는 동사 중심 이름을 사용한다.
- 불리언 반환 메서드는 `is_`, `has_`, `can_` 접두사를 우선 고려한다.

예시:

- `sweep_once()`
- `get_expiration()`
- `is_running()`
- `calculate_remaining_seconds()`

### 3-4. 변수명

- 변수명은 `snake_case`를 사용한다.
- 의미 없는 짧은 이름은 반복문 인덱스처럼 범위가 아주 작은 경우에만 사용한다.
- 시간 값은 `now`, `expires_at`, `remaining_seconds`처럼 의미가 드러나야 한다.

### 3-5. 상수명

- 상수는 `UPPER_SNAKE_CASE`를 사용한다.
- 프로토콜 에러 메시지, 기본 제한값, 종료 코드는 가능한 한 상수로 분리한다.

예시:

- `DEFAULT_PORT`
- `MAX_REQUEST_SIZE`
- `ERR_INVALID_REQUEST`

### 3-6. 테스트명

- 테스트 파일명은 `tests/test_<target>.py` 규칙을 사용한다.
- 테스트 함수명은 `test_<condition>_<expected_result>()` 형태를 권장한다.

예시:

- `tests/test_request_decoder.py`
- `test_get_returns_null_when_key_is_missing()`

## 4. 계층별 코딩 규칙

### 4-1. `cmd`

- 실행 진입점만 담당한다.
- 설정 로딩과 객체 조립은 가능하지만 비즈니스 로직을 두지 않는다.
- 서버 내부 구현 세부사항을 직접 처리하지 않는다.

### 4-2. `internal/protocol/resp`

- RESP3 프레임 파싱과 직렬화만 담당한다.
- 저장소를 직접 참조하지 않는다.
- 비즈니스 규칙을 판단하지 않는다.

### 4-3. `internal/command`

- 내부 명령 모델, 파싱, 검증만 담당한다.
- 소켓 처리나 저장소 접근을 하지 않는다.
- 서비스 실행을 직접 호출하지 않는다.

### 4-4. `internal/service`

- 명령별 비즈니스 규칙을 담당한다.
- 저장소와 만료 처리 계층을 조합할 수 있다.
- RESP 프레임 형식이나 소켓 송수신을 알면 안 된다.

### 4-5. `internal/repository`

- 저장과 조회만 담당한다.
- TTL 계산, 비즈니스 규칙, 네트워크 처리를 하지 않는다.
- `service`를 참조하면 안 된다.

### 4-6. `internal/expiration`

- TTL 계산, 만료 정리, 백그라운드 sweep만 담당한다.
- RESP 응답 형식과 직접 결합하지 않는다.
- 명령 종류를 직접 해석하지 않는다.

### 4-7. `internal/server`

- 연결 수락, 세션 처리, graceful shutdown만 담당한다.
- 비즈니스 규칙은 `service`에 위임한다.
- 저장소를 직접 호출하지 않는다.

### 4-8. `internal/guard`

- 연결 수, 요청 크기, 파싱 제한 같은 자원 보호 정책만 담당한다.
- 명령 실행 결과를 바꾸지 않는다.

### 4-9. `internal/observability`

- 로그와 메트릭 기록만 담당한다.
- 로그를 위해 비즈니스 로직을 바꾸지 않는다.

## 5. 에러 및 응답 처리 규칙

### 5-1. 에러 메시지 규칙

- RESP3 오류 메시지는 가능한 한 `internal/protocol/resp/messages.py`에 모은다.
- 문자열 리터럴을 여러 파일에 중복 작성하지 않는다.
- 같은 오류 상황은 같은 메시지를 사용한다.

### 5-2. 예외 사용 규칙

- 정상적인 명령 실패는 예외보다 명시적인 결과 또는 에러 모델로 표현한다.
- 예외는 주로 프로그래밍 오류, 예상하지 못한 내부 오류, 시스템 오류에 사용한다.
- 세션 처리 계층은 예외를 잡아 요청 단위로 격리해야 한다.

### 5-3. 반환 규칙

- `service`는 RESP 프레임 자체보다 도메인 결과를 반환하는 쪽을 우선한다.
- `protocol` 계층이 최종 RESP3 표현을 담당한다.
- `CLI`는 서버 응답을 해석해 출력 형식과 종료 코드를 결정한다.

## 6. 테스트 규칙

### 6-1. 테스트 분류

- 단위 테스트: 한 모듈의 책임만 검증한다.
- 통합 테스트: 여러 계층이 연결된 흐름을 검증한다.
- end-to-end 테스트: CLI에서 서버까지 실제 사용 흐름을 검증한다.

### 6-2. 단위 테스트 대상

- `internal/command`
- `internal/protocol/resp`
- `internal/service`
- `internal/expiration`
- `internal/guard`

### 6-3. 통합 테스트 대상

- `internal/server/session_handler.py`
- `cmd/mini_redis_cli/main.py`
- `HELLO 3 -> 명령 실행 -> RESP3 응답` 전체 흐름

### 6-4. 테스트 작성 규칙

- 새로운 기능을 추가하면 해당 기능의 단위 테스트를 함께 추가한다.
- 버그를 수정하면 재현 테스트를 먼저 추가하는 것을 원칙으로 한다.
- 테스트 이름만 보고 무엇을 검증하는지 알 수 있어야 한다.

## 8. 협업 절차 규칙

### 8-1. 파일 소유권

- 담당자는 자기 소유 파일을 우선 수정한다.
- 공동 조정 파일은 최종 소유 담당자가 병합 책임을 가진다.
- 다른 담당자의 소유 파일을 수정해야 하면 먼저 합의한다.

### 8-2. 공동 조정 파일

다음 파일은 충돌 가능성이 높으므로 주의한다.

- `internal/service/command_service.py`
- `internal/server/session_handler.py`
- `cmd/mini_redis_server/main.py`
- `cmd/mini_redis_cli/main.py`
- `internal/config/runtime_config.py`

### 8-3. 병합 순서

- 프로토콜 및 명령 인터페이스를 먼저 고정한다.
- 저장소와 서비스 구현을 그 위에 맞춘다.
- 서버와 CLI는 마지막에 조립한다.
- 통합 테스트는 전체 경로가 연결된 뒤 보강한다.

## 9. Git 브랜치 관리 규칙

### 9-1. 기본 브랜치 전략

- 기본 브랜치는 `main`과 `dev`를 사용한다.
- `main`은 안정 상태를 유지하는 브랜치로 사용한다.
- `dev`는 팀의 통합 개발 브랜치로 사용한다.
- 각 개발자는 개별 작업을 위한 브랜치를 생성한 뒤 `dev`로 병합한다.

### 9-2. 권장 작업 흐름

1. 작업 시작 전에 이슈를 생성한다.
2. 이슈 범위를 확인하고 담당자를 지정한다.
3. 이슈 단위로 작업 브랜치를 만든다.
4. 작업 완료 후 `dev`를 대상으로 PR 또는 병합을 진행한다.
5. 통합 검증이 끝난 뒤 `dev`를 `main`에 병합한다.

### 9-3. 브랜치 역할

- `main`
  - 배포 가능하거나 기준 상태로 간주하는 브랜치
  - 직접 작업하지 않는다.
- `dev`
  - 팀 공통 개발 브랜치
  - 각 작업 브랜치의 통합 지점
- 작업 브랜치
  - 기능, 버그 수정, 테스트, 문서 수정 등 개별 작업 단위 브랜치

### 9-4. 브랜치 네이밍 규칙

- 브랜치명은 작업 종류와 범위를 드러내야 한다.
- 권장 형식은 `<type>/<issue-id>-<short-description>`이다.

예시:

- `feat/12-request-decoder`
- `feat/18-expire-ttl-service`
- `fix/27-session-handler-error-path`
- `test/31-cli-integration`
- `docs/35-collaboration-rules`

### 9-5. 이슈 생성 원칙

- 작업 브랜치는 가능한 한 이슈와 1:1로 대응시킨다.
- 하나의 이슈에는 하나의 명확한 목표만 둔다.
- 너무 큰 작업은 여러 이슈로 나눈다.
- 공동 조정 파일 수정은 별도 이슈로 분리하는 것을 권장한다.

### 9-6. `dev` 병합 규칙

- 작업 브랜치는 직접 `main`에 병합하지 않는다.
- 먼저 `dev`에 병합해 통합 검증을 거친다.
- `dev` 병합 전에는 최소한 담당 범위 테스트 또는 수동 검증 결과를 남긴다.
- 충돌 가능성이 큰 파일은 병합 전에 담당자 간 재확인한다.

### 9-7. `main` 병합 규칙

- `main` 병합은 팀 공통 검증이 끝난 뒤 수행한다.
- 최소 기준은 다음과 같다.
  - 필수 명령 흐름이 동작한다.
  - `HELLO 3` 협상이 동작한다.
  - CLI와 서버의 기본 통신이 동작한다.
  - 치명적인 통합 충돌이 없다.

## 10. 코드 품질 규칙

### 9-1. 타입 힌트

- 공개 함수와 메서드에는 타입 힌트를 작성한다.
- 반환 타입을 생략하지 않는다.
- 테스트 코드에서도 타입 힌트가 과도하게 방해되지 않는 범위에서 유지한다.

### 9-2. 주석

- 주석은 필요한 곳에만 짧게 작성한다.
- 코드가 무엇을 하는지보다 왜 그렇게 하는지를 설명한다.
- 문서에 이미 있는 내용을 장황하게 반복하지 않는다.

### 9-3. 로깅

- 로그는 구조화된 필드 중심으로 남긴다.
- 에러 로그에는 최소한 요청 종류, 실패 지점, 예외 요약이 있어야 한다.
- 정상 흐름 로그는 과도하게 상세하지 않게 유지한다.

### 9-4. 포맷

- 한 파일 안에서는 같은 스타일을 유지한다.
- 불필요한 공백, 불필요한 줄바꿈, 섞인 네이밍 스타일을 허용하지 않는다.
- 자동 포맷 도구를 도입한다면 팀 전체가 같은 도구와 같은 설정을 사용한다.

## 11. 권장 체크리스트

코드 제출 전에 다음을 확인하는 것을 권장한다.

- 네이밍 규칙을 지켰는가
- 계층 경계를 넘지 않았는가
- 에러 메시지를 중복 정의하지 않았는가
- 테스트를 추가하거나 갱신했는가
- 관련 문서를 함께 수정했는가
- 공동 조정 파일 수정 시 담당자와 합의했는가

이 문서는 팀이 병렬로 작업하는 동안 코드 스타일과 협업 방식의 기준 문서로 사용한다.
