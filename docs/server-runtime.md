# 미니 Redis 서버 운영 설정 문서

## 1. 문서 목적

이 문서는 미니 Redis 서버와 CLI의 기본 운영 설정값을 정의한다.

본 문서는 [`requirements.md`](requirements.md), [`architecture.md`](architecture.md), [`api-spec.md`](api-spec.md)를 보완하며, 실제 실행 시 필요한 포트, 타임아웃, 제한값, 종료 정책의 기본값을 명시하는 것을 목적으로 한다.

## 2. 적용 범위

- 서버 기본 바인드 설정
- CLI 기본 접속 설정
- 연결 및 요청 타임아웃
- 동시 연결 수 및 요청 크기 제한
- RESP 프레임 파싱 제한
- graceful shutdown 정책
- 기본 로그 및 메트릭 운영값

## 3. 기본 네트워크 설정

### 3-1. 서버 기본 바인드 주소

- 기본 바인드 주소: `127.0.0.1`
- 기본 포트: `6379`

설명:

- 1차 구현에서는 로컬 개발과 테스트를 기본 목적으로 하므로 loopback 주소를 기본값으로 사용한다.
- 운영 환경에서는 명시적 설정을 통해 다른 인터페이스 주소로 변경할 수 있어야 한다.

### 3-2. CLI 기본 접속 설정

- 기본 접속 주소: `127.0.0.1`
- 기본 접속 포트: `6379`

설명:

- CLI는 별도 인자가 없으면 위 기본값으로 서버 연결을 시도한다.
- CLI는 `--host`, `--port` 인자를 통해 값을 덮어쓸 수 있어야 한다.

## 4. 타임아웃 설정

### 4-1. 연결 타임아웃

- 기본값: `3초`

설명:

- CLI가 서버에 접속을 시도할 때 3초 내 연결되지 않으면 연결 실패로 간주한다.

### 4-2. 서버 소켓 읽기 타임아웃

- 기본값: `5초`

설명:

- 클라이언트가 연결 후 요청을 보내지 않으면 서버는 유휴 또는 비정상 연결로 간주하고 종료할 수 있다.

### 4-3. 서버 응답 송신 타임아웃

- 기본값: `5초`

설명:

- 서버가 응답을 전송하지 못하는 상태가 지속되면 연결을 종료할 수 있다.

### 4-4. 유휴 연결 타임아웃

- 기본값: `5초`

설명:

- 1차 구현은 요청 1개 처리 후 응답 1개를 반환하고 연결 종료를 기본으로 하므로 긴 유휴 연결을 유지하지 않는다.

## 5. 연결 및 요청 제한값

### 5-1. 최대 동시 연결 수

- 기본값: `100`

설명:

- 1차 구현에서는 과도한 연결로 인한 자원 고갈을 방지하기 위해 상한을 둔다.
- 상한 초과 시 서버는 새 연결을 거부하거나 즉시 종료할 수 있다.

### 5-2. 최대 요청 크기

- 기본값: `4KB`

설명:

- RESP3 프레임 전체 크기 기준으로 제한한다.
- 제한 초과 시 서버는 요청을 거부하고 오류 응답 또는 연결 종료를 수행할 수 있다.

### 5-3. 최대 배열 원소 수

- 기본값: `64`

설명:

- RESP Array 요청에서 허용할 최대 원소 수를 제한한다.
- 선택 기능과 향후 확장 자료구조를 고려해 여유 있는 기본값을 둔다.

### 5-4. 최대 RESP 중첩 깊이

- 기본값: `8`

설명:

- 과도한 중첩 프레임으로 인한 파서 자원 고갈을 방지한다.

### 5-5. 최대 Blob String 길이

- 기본값: `4KB`

설명:

- 단일 문자열 값과 단일 인자의 최대 크기를 제한한다.
- 향후 운영 환경에서는 메모리와 워크로드에 맞게 조정해야 한다.

### 5-6. 만료 정리 주기

- 기본값: `10초`

설명:

- 백그라운드 만료 정리 작업이 TTL 저장소를 검사하는 주기이다.

### 5-7. 만료 정리 배치 크기

- 기본값: `100`

설명:

- 한 번의 sweep에서 검사 또는 정리할 최대 키 수이다.

### 5-8. 만료 정리 활성화 여부

- 기본값: `활성화`

설명:

- 1차 구현은 지연 삭제와 함께 주기적 정리를 기본 활성화 상태로 둔다.

## 6. 요청 처리 정책

### 6-1. 처리 모델

- 기본 정책: `복수 연결 허용, 명령 실행은 순차 처리`

설명:

- 연결은 여러 개 받을 수 있으나, 저장소 변경은 직렬 처리하는 것을 기본으로 한다.
- 이는 구현 단순성과 데이터 일관성 확보를 위한 초기 운영 정책이다.

### 6-2. 요청당 연결 정책

- 기본 정책: `연결당 요청 1개, 응답 1개 후 종료`

설명:

- 1차 구현에서는 세션 지속보다 단순성과 장애 격리를 우선한다.

## 7. 종료 및 복구 정책

### 7-1. Graceful Shutdown 대기 시간

- 기본값: `10초`

설명:

- 종료 신호를 받으면 새 연결 수락을 중단하고, 진행 중 요청 정리를 최대 10초까지 대기한 후 종료한다.

### 7-2. 비정상 종료 시 정책

- 기본 정책: `인메모리 데이터는 복구하지 않는다`

설명:

- 현재 범위에서는 영속성이 없으므로 프로세스 비정상 종료 후 데이터 복구는 지원하지 않는다.
- 향후 영속성 도입 시 스냅샷 또는 로그 기반 복구 전략을 별도 정의해야 한다.

## 8. 로그 및 메트릭 설정

### 8-1. 기본 로그 레벨

- 기본값: `INFO`

설명:

- 정상 시작/종료, 연결 수락, 명령 처리 성공/실패, 내부 오류를 기록한다.

### 8-2. 디버그 로그 레벨

- 선택값: `DEBUG`

설명:

- 개발 및 장애 분석 시 요청 흐름, 파싱 결과, 검증 실패 원인까지 더 자세히 기록할 수 있다.

### 8-3. 기본 메트릭 항목

- 총 연결 수
- 현재 활성 연결 수
- 총 요청 수
- 명령별 요청 수
- 오류 응답 수
- 내부 오류 수
- 자원 제한 초과 횟수

## 9. CLI 실행 기본값

### 9-1. 기본 명령 형식

```text
mini-redis-cli --host 127.0.0.1 --port 6379 GET mykey
```

### 9-2. CLI 종료 코드 정책

- 성공 시: `0`
- 사용법 오류 시: `2`
- 연결 실패 또는 통신 실패 시: `3`
- 서버가 오류 응답을 반환한 경우: `4`

설명:

- 종료 코드는 자동화 스크립트와 테스트에서 활용할 수 있어야 한다.

## 11. 프로덕션 확장 시 권장 사항

- 운영 환경에서는 기본 바인드 주소를 명시적으로 설정해야 한다.
- 연결 수 상한과 요청 크기 상한은 서버 메모리와 예상 트래픽에 맞게 조정해야 한다.
- 장기적으로는 헬스 체크, 메트릭 노출, 영속성, 복제, 샤딩, 인증 기능을 단계적으로 추가해야 한다.
- 단일 프로세스 기반 운영에서 한계를 보이면 이벤트 루프 또는 멀티 워커 모델을 검토해야 한다.

## 12. 문서 적용 메모

- 본 문서의 포트, 타임아웃, 제한값은 1차 구현 기본값으로 확정된 값이다.
- 구현체는 이 값을 그대로 사용할 수 있어야 하며, 운영 환경에서는 설정 외부화를 통해 조정할 수 있어야 한다.
- 요구사항 문서의 추후 상세화 항목은 이 값 자체가 아니라 환경별 재조정 방식과 세부 운영 정책을 의미한다.
