# 미니 Redis 요구사항 정의서

## 1. 문서 목적

이 문서는 학습용 미니 Redis 구현이 반드시 만족해야 하는 기능 요구사항과 범위를 정의한다.

이 문서의 목적은 다음과 같다.

- 구현 범위를 명확히 제한한다.
- 명령별 기대 동작을 고정한다.
- 테스트 기준과 수용 기준을 제공한다.
- 설계 및 구현 단계에서 해석 차이를 줄인다.

## 2. 프로젝트 범위

본 프로젝트는 Redis의 모든 기능을 재현하는 것이 아니라, 핵심 개념 학습을 위한 최소 기능의 키-값 저장소 서버와 CLI 클라이언트를 구현하는 것을 목표로 한다.

프로젝트의 핵심 학습 범위는 다음과 같다.

- 문자열 기반 키-값 저장
- 명령 기반 인터페이스 처리
- TTL 기반 만료 처리
- TCP 소켓 기반 서버 연결 처리
- CLI 기반 클라이언트 실행 및 명령 전송
- 기본적인 예외 및 경계 조건 처리

## 3. 전제 및 제약 사항

- 본 프로젝트는 학습용 구현을 전제로 한다.
- 본 프로젝트는 단일 프로세스 환경을 기준으로 한다.
- 데이터는 메모리 내에서 관리한다.
- 프로세스 종료 후 데이터 영속성은 제공하지 않는다.
- 분산 처리, 복제, 클러스터링은 지원하지 않는다.
- TCP 소켓 통신은 지원 범위에 포함한다.
- 1차 구현부터 지원 명령 집합에 한정해 Redis RESP3 요청/응답 규약과 호환되는 것을 목표로 한다.
- 서버는 `HELLO 3` 협상을 지원해야 하며, 1차 구현의 CLI는 세션 시작 시 `HELLO 3`을 먼저 전송해야 한다.

## 4. 기능 범위

### 4-1. 필수 기능

다음 기능은 최초 구현에서 반드시 지원해야 한다.

- TCP 소켓 기반 서버 실행
- CLI 기반 클라이언트 실행
- 클라이언트와 서버 간 명령 송수신
- RESP3 요청 파싱
- RESP3 응답 직렬화
- `HELLO 3` 협상 처리
- 문자열 키 저장
- 문자열 값 저장
- `SET key value`
- `GET key`
- `DEL key`
- `EXPIRE key seconds`
- `TTL key`

### 4-2. 선택 기능

다음 기능은 필수 범위에 포함하지 않으며, 추후 확장 기능으로 고려할 수 있다.

- `SET key value EX seconds`
- `EXISTS key`
- `KEYS pattern`

### 4-3. 2차 확장 자료구조 로드맵

1차 구현은 문자열 자료형만 지원한다.

이후 확장 로드맵은 다음 순서를 권장한다.

1. `Hash`
2. `List`
3. `Set`
4. `Sorted Set`

설명:

- `Hash`는 필드-값 구조를 통해 자료형 확장의 첫 단계로 적합하다.
- `List`는 순서 기반 자료구조와 양방향 연산을 도입하기 좋다.
- `Set`은 중복 없는 집합 연산을 다루는 데 적합하다.
- `Sorted Set`은 점수 기반 정렬과 더 복잡한 내부 자료구조가 필요하므로 후순위가 적절하다.

## 5. 데이터 요구사항

- 키는 문자열이어야 한다.
- 값은 문자열이어야 한다.
- 각 키는 최대 1개의 값만 가진다.
- 각 키는 최대 1개의 TTL 정보만 가진다.
- 동일한 키에 대해 새 값을 저장하면 기존 값을 대체한다.

## 6. 명령 처리 공통 요구사항

- 시스템은 클라이언트가 전송한 RESP3 요청을 서버에서 받아 처리할 수 있어야 한다.
- 명령어는 RESP3 Array의 첫 번째 원소를 기준으로 식별해야 한다.
- 명령어별 필수 인자 개수를 검증해야 한다.
- 잘못된 명령 형식에 대해 일관된 오류 응답을 반환해야 한다.
- 지원하지 않는 명령에 대해 일관된 오류 응답을 반환해야 한다.
- 명령 처리 결과는 테스트 가능한 형태의 명확한 응답이어야 한다.

## 7. 서버 및 CLI 요구사항

### 7-1. 서버 요구사항

- 시스템은 지정된 포트에서 클라이언트 연결을 받을 수 있어야 한다.
- 시스템은 TCP 소켓 기반으로 클라이언트 요청을 수신할 수 있어야 한다.
- 서버는 수신한 요청을 명령 처리 계층으로 전달할 수 있어야 한다.
- 서버는 처리 결과를 클라이언트에 다시 전송할 수 있어야 한다.
- 서버는 잘못된 요청 형식에 대해 일관된 오류 응답을 반환해야 한다.
- 서버는 최소 1개 이상의 클라이언트 연결을 처리할 수 있어야 한다.
- 서버는 복수 클라이언트의 동시 연결을 허용할 수 있어야 하며, 명령 실행은 순차 처리 정책을 기본으로 적용할 수 있다.
- 서버는 RESP3 프레임을 해석할 수 있어야 한다.
- 서버는 `HELLO 3` 요청을 처리하고 RESP3 세션을 성립시킬 수 있어야 한다.
- 서버는 지원 명령에 대해 RESP3 타입으로 응답해야 한다.

### 7-2. CLI 요구사항

- 시스템은 사용자가 명령행에서 실행할 수 있는 CLI를 제공해야 한다.
- CLI는 서버 주소와 포트 정보를 받아 서버에 연결할 수 있어야 한다.
- CLI는 사용자가 입력한 명령을 서버로 전송할 수 있어야 한다.
- CLI는 서버 응답을 사용자에게 표시할 수 있어야 한다.
- CLI는 연결 실패, 타임아웃, 잘못된 사용법에 대해 적절한 오류 메시지를 표시해야 한다.
- CLI는 단일 명령 실행 방식과 대화형 REPL 실행 방식을 모두 지원해야 한다.
- 대화형 REPL에서는 여러 명령을 연속으로 입력하고 결과를 순차적으로 확인할 수 있어야 한다.
- CLI는 서버와 RESP3 기반으로 통신해야 한다.
- CLI는 세션 시작 시 `HELLO 3`을 먼저 전송해야 한다.
- CLI는 성공 응답과 실패 응답을 구분해 사용자에게 표시할 수 있어야 한다.
- CLI는 성공/실패 여부를 종료 코드로도 구분할 수 있어야 한다.

## 8. 명령별 요구사항

### 8-1. SET

형식:
`SET key value`

요구사항:

- 시스템은 지정한 키에 값을 저장해야 한다.
- 동일한 키가 이미 존재하면 기존 값을 새 값으로 덮어써야 한다.
- 기존 키에 TTL이 설정되어 있었다면, `SET` 수행 후 기존 TTL은 제거되어야 한다.
- `SET`은 성공 여부를 나타내는 일관된 성공 응답을 반환해야 한다.
- 인자 수가 맞지 않으면 오류 응답을 반환해야 한다.

### 8-2. GET

형식:
`GET key`

요구사항:

- 시스템은 지정한 키의 값을 조회해야 한다.
- 키가 존재하고 만료되지 않았다면 저장된 값을 반환해야 한다.
- 키가 존재하지 않으면 null 성격의 응답을 반환해야 한다.
- 키가 만료되었다면 해당 키를 제거한 뒤 null 성격의 응답을 반환해야 한다.
- 인자 수가 맞지 않으면 오류 응답을 반환해야 한다.

### 8-3. DEL

형식:
`DEL key`

요구사항:

- 시스템은 지정한 키를 데이터 저장소에서 삭제해야 한다.
- 해당 키의 TTL 정보가 존재하면 함께 삭제해야 한다.
- 삭제 대상 키가 존재하면 성공 삭제 응답을 반환해야 한다.
- 삭제 대상 키가 존재하지 않으면 실패 또는 0 성격의 응답을 반환해야 한다.
- 키가 만료된 상태였다면 만료 정리 후 존재하지 않는 키로 간주해야 한다.
- 인자 수가 맞지 않으면 오류 응답을 반환해야 한다.

### 8-4. EXPIRE

형식:
`EXPIRE key seconds`

요구사항:

- 시스템은 지정한 키가 존재할 때만 TTL을 설정해야 한다.
- `seconds`는 초 단위의 정수여야 한다.
- 키가 존재하지 않으면 TTL을 설정하지 않아야 한다.
- 키가 존재하지 않을 경우 실패 또는 0 성격의 응답을 반환해야 한다.
- 키가 존재할 경우 TTL 설정 성공 응답을 반환해야 한다.
- `seconds`가 0 이하인 경우 오류 응답을 반환해야 한다.
- 인자 수가 맞지 않거나 `seconds`가 정수가 아니면 오류 응답을 반환해야 한다.

### 8-5. TTL

형식:
`TTL key`

요구사항:

- 시스템은 지정한 키의 남은 TTL을 초 단위로 반환해야 한다.
- 키가 존재하지 않으면 미존재를 나타내는 지정 응답을 반환해야 한다.
- 키가 존재하지만 TTL이 설정되어 있지 않으면 별도 지정 응답을 반환해야 한다.
- 키가 만료되었다면 해당 키를 제거한 뒤 미존재 응답을 반환해야 한다.
- 키가 유효하고 TTL이 존재하면 남은 시간을 초 단위 정수로 반환해야 한다.
- 인자 수가 맞지 않으면 오류 응답을 반환해야 한다.

## 9. TTL 정책 요구사항

- TTL은 키 단위로 관리되어야 한다.
- TTL은 현재 시각을 기준으로 계산되어야 한다.
- TTL이 설정된 키는 만료 시점 이후 접근 시 삭제 처리되어야 한다.
- 만료 검사는 최소한 `GET`, `DEL`, `EXPIRE`, `TTL` 수행 시 적용되어야 한다.
- 만료된 키는 값 저장소와 TTL 저장소에서 모두 제거되어야 한다.
- TTL 계산 기준은 구현과 테스트에서 일관되어야 한다.
- TTL 반환 시 소수점은 허용하지 않으며 정수 초 단위만 사용해야 한다.
- 1차 구현은 접근 시 만료를 처리하는 지연 삭제 방식을 지원해야 한다.
- 1차 구현은 백그라운드 주기적 정리를 통해 접근이 없는 만료 키도 메모리에서 제거할 수 있어야 한다.
- 지연 삭제는 정확성 보장을 위한 기본 메커니즘으로 유지되어야 한다.
- 주기적 정리는 메모리 회수 효율을 보완하는 보조 메커니즘으로 동작해야 한다.

## 10. 입력 및 오류 처리 요구사항

- 명령어가 비어 있거나 공백만 포함된 경우 오류 응답을 반환해야 한다.
- 지원하지 않는 명령어는 오류 응답을 반환해야 한다.
- 인자 개수가 부족하거나 초과되면 오류 응답을 반환해야 한다.
- 정수 인자가 필요한 위치에 정수가 아닌 값이 오면 오류 응답을 반환해야 한다.
- 오류 응답 형식은 명령 종류와 관계없이 일관되어야 한다.
- 서버 연결 요청이 유효하지 않으면 연결 또는 요청 단계에서 적절한 오류를 반환해야 한다.
- CLI 인자가 유효하지 않으면 사용법 또는 오류 메시지를 출력해야 한다.
- RESP3 프레임이 손상되었거나 지원하지 않는 타입이 들어오면 프로토콜 오류 응답 또는 연결 종료 정책을 적용해야 한다.
- CLI는 RESP3 오류 응답을 실패로 해석해야 한다.

## 11. 비기능 요구사항

- 구현은 이해하기 쉽고 학습 목적에 적합해야 한다.
- 핵심 로직은 테스트 가능하도록 분리되어야 한다.
- 만료 처리 로직은 중복 없이 재사용 가능해야 한다.
- 명령 처리 로직과 저장소 관리 로직은 분리되는 것이 바람직하다.
- 필수 기능에 대해 자동화된 테스트가 가능해야 한다.
- 서버 연결 처리 로직과 명령 실행 로직은 분리되는 것이 바람직하다.
- CLI는 서버 없이 비즈니스 로직을 직접 호출하지 않아야 한다.

## 12. 운영 및 장애 대응 요구사항

### 12-1. 현재 구현에서 반드시 고려할 항목

- 서버는 잘못된 요청 1건이 전체 프로세스를 중단시키지 않도록 요청 단위로 오류를 격리해야 한다.
- 서버는 내부 오류 발생 시 프로세스 비정상 종료 대신 일관된 오류 응답 또는 연결 종료 정책을 적용해야 한다.
- 서버는 연결 수, 요청 수, 오류 수 등 기본 운영 상태를 기록할 수 있어야 한다.
- 서버는 정상 종료 시 열린 리소스를 정리할 수 있어야 한다.
- 타임아웃 또는 유휴 연결 정책을 둘 수 있어야 하며, 장시간 점유 연결이 전체 서버를 마비시키지 않도록 설계되어야 한다.
- 백그라운드 만료 정리 작업은 서버 종료 시 안전하게 중지되어야 한다.
- 백그라운드 만료 정리 작업은 주기와 처리량을 제어할 수 있어야 한다.

### 12-2. 프로덕션 수준으로 확장 시 필요한 항목

- 장애 원인 분석을 위한 구조화 로그와 메트릭 수집 기능이 필요하다.
- 헬스 체크 및 준비 상태 확인을 위한 운영 인터페이스가 필요하다.
- 메모리 사용량 제한, 연결 수 제한, 요청 크기 제한 등 자원 보호 정책이 필요하다.
- 재시작 후 데이터 복구를 위한 영속성 전략이 필요하다.
- 단일 노드 장애 대응을 위한 복제 및 장애 조치 전략이 필요하다.
- 트래픽 증가 대응을 위한 샤딩 또는 클러스터링 전략이 필요하다.
- 무중단 배포와 graceful shutdown을 지원하는 운영 절차가 필요하다.

## 13. 제외 범위

다음 항목은 현재 요구사항 범위에 포함하지 않는다.

- 영속성 저장
- 복제 및 클러스터링
- Pub/Sub
- 트랜잭션
- 리스트, 셋, 해시 등 문자열 외 자료형
- 정렬 셋과 기타 복합 자료구조
- Lua 스크립팅
- 인증 및 권한 관리
- 고성능 최적화

단, 위 항목들은 프로덕션 수준 확장을 위한 고려 대상이며 장기적으로는 아키텍처 확장 포인트에 반영되어야 한다.

## 14. 수용 기준

다음 조건을 만족하면 1차 구현이 요구사항을 충족한 것으로 본다.

- 서버가 지정 포트에서 실행되고 클라이언트 연결을 수락한다.
- CLI가 서버에 연결해 명령을 전송할 수 있다.
- 서버가 명령을 처리하고 응답을 반환한다.
- CLI가 `HELLO 3`을 전송하고 서버가 RESP3 세션을 정상 성립시킨다.
- RESP3 요청과 응답 형식이 명세와 일치한다.
- `SET`, `GET`, `DEL`, `EXPIRE`, `TTL`이 요구된 형식대로 동작한다.
- 만료된 키는 접근 시 자동으로 정리된다.
- 잘못된 입력에 대해 일관된 오류 응답을 반환한다.
- 필수 기능에 대한 자동화 테스트가 존재한다.
- 테스트를 통해 주요 경계 조건이 검증된다.
- 잘못된 요청 또는 내부 처리 오류가 전체 서버 중단으로 이어지지 않는다.

## 15. 추후 추가 상세화 가능 항목

다음 항목은 기본 정책과 기본값은 정해졌지만, 구현 단계에서 더 구체화할 수 있다.

- 운영 환경용 설정 파일, 환경 변수, 실행 인자 우선순위
- 향후 영속성 도입 시 스냅샷 또는 로그 기반 복구 정책
- RESP3 서버 푸시 메시지 지원 여부
- 자원 제한 초과 시 오류 응답과 연결 종료의 선택 기준
